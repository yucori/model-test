## 1. RAG 파이프라인 구조

RAG(Retrieval-Augmented Generation)는 단일 모델이 아니라 **여러 단계가 직렬로 연결된 파이프라인**입니다.
각 단계가 독립적으로 실패할 수 있으므로, 전체 품질은 가장 약한 단계에 의해 결정됩니다.

```
문서 파일
   │
   ▼
① 파싱 (PDF/DOCX → 텍스트)
   │  document_processor.py
   ▼
② 청킹 (텍스트 → 청크 목록)
   │  chunk_text(chunk_size, overlap)
   ▼
③ 임베딩 & 저장 (청크 → 벡터 → ChromaDB)
   │  vector_store.add_chunks()
   │
   │          ← 질문 입력
   ▼
③ 검색 (질문 → 유사 청크 top-k)
   │  vector_store.query()
   ▼
④ 컨텍스트 조립 (청크 → 프롬프트)
   │  test_runner.py
   ▼
⑤ LLM 생성 (프롬프트 → 답변)
   │  llm_clients.call_model()
   ▼
⑥ 평가 (답변 → 0~10점)
      evaluator.py
```

### 핵심 원칙

> **"나쁜 입력은 좋은 출력을 만들 수 없다"**

- ① 파싱이 깨지면 ② 청킹도 깨지고, 이후 모든 단계가 무너집니다.
- ③ 검색이 엉뚱한 청크를 반환하면 LLM이 아무리 좋아도 잘못된 답변을 생성합니다.
- ⑤ LLM 프롬프트가 불명확하면 좋은 검색 결과도 활용되지 않습니다.

이 시스템에서 **임베딩 탭(② 진단)**과 **OCR 탭(① 진단)**을 먼저 사용해
상위 단계 문제를 해결한 뒤 LLM 비교로 넘어가는 순서를 권장합니다.

## 2. ① 문서 파싱 품질

### 현재 구현

`backend/app/services/document_processor.py`

- **PDF**: pdfplumber로 텍스트 레이어 추출 → 페이지당 50자 미만이면 easyOCR(한국어+영어)로 폴백
- **DOCX**: python-docx로 단락 + 표 셀 추출

### 확인 방법

실행 탭 > **① OCR 검증**에서 각 문서의 원문 미리보기와 청크를 직접 확인합니다.

### 흔한 문제와 원인

| 증상 | 원인 | 확인 위치 |
|------|------|-----------|
| "페이지 1", "페이지 2" 텍스트가 청크마다 반복 | 푸터 제거 미처리 | OCR 탭 > 원문 미리보기 |
| 표 데이터가 "배13일" 같이 붙어서 출력 | 표 셀 파싱 오류 | OCR 탭 > 원문 미리보기 |
| 깨진 글자 ("ㄹ빽ㅗ") | 스캔 해상도 낮음 / OCR 언어 미스매치 | OCR 탭 |
| 같은 문장이 청크에 중복 등장 | 오버랩 과다 or 문서 자체 중복 | OCR 탭 > 청크 목록 |
| 섹션 번호만 있는 짧은 청크 | 청크 크기보다 단락이 짧음 | OCR 탭 > 청크 목록 |

### 개선 방법

**헤더/푸터 제거** (`document_processor.py` → `_extract_pdf` 후처리):

```python
import re

def _clean_text(text: str) -> str:
    # 페이지 번호 패턴 제거
    text = re.sub(r'\n페이지\s*\d+\s*\n', '\n', text)
    text = re.sub(r'\n\d+\s*/\s*\d+\s*\n', '\n', text)   # "3 / 10"
    text = re.sub(r'\n-\s*\d+\s*-\s*\n', '\n', text)     # "- 3 -"
    # 연속 공백/줄바꿈 정리
    text = re.sub(r'\n{3,}', '\n\n', text)
    return text.strip()
```

**OCR 해상도 향상** (스캔 PDF 품질 개선):

```python
# 현재: 2× 배율
mat = fitz.Matrix(2, 2)

# 개선: 3× 배율 (품질↑, 처리속도↓)
mat = fitz.Matrix(3, 3)
```

**품질 기준선**: OCR 탭에서 문서 원문을 펼쳤을 때 일반적인 한국어 문장이 자연스럽게 읽혀야 합니다.
깨진 글자나 의미 없는 기호가 많다면 파싱 단계부터 수정이 필요합니다.

## 3. ② 청킹 전략

### 청킹의 목표

> **청크 하나가 하나의 완결된 답변 단위를 담아야 합니다.**

LLM에게 전달되는 컨텍스트는 top-k개의 청크입니다.
청크 경계가 중요한 정보를 잘라버리면 아무리 검색이 잘 돼도 LLM이 불완전한 정보로 답변합니다.

```
나쁜 청크 경계:                    좋은 청크 경계:
────────────────────               ────────────────────
3. 환불 정책                       3. 환불 정책
구매 후 7일 이내 환불              구매 후 7일 이내 환불 가능.
가능합니다. 단, 신선식품            단, 신선식품·냉동식품은 수령 후
────────────────────               24시간 이내 신청 필요.
← 다음 청크로 잘림                 포장 미개봉 상태여야 합니다.
                                   ────────────────────
                                   ← 정책 내용이 완결됨
```

### 파라미터 가이드

현재 시스템: 준비 탭 > 청킹 전략 슬라이더, OCR 탭 > 실시간 미리보기

| 설정 | chunk_size | overlap | 적합한 상황 |
|------|-----------|---------|-------------|
| 소형 | 300자 | 50자 | 짧은 Q&A, 단답형 정책 항목 |
| 표준 | 600자 | 100자 | 일반 CS 정책 문서 (권장 시작점) |
| 대형 | 1000자 | 150자 | 절차 설명, 긴 배경 설명 문서 |

**오버랩의 역할**: 청크 A의 끝 부분이 청크 B의 앞에도 포함되어, 경계에 걸친 내용이 검색에서 누락되지 않도록 합니다.
오버랩이 너무 크면 청크 수가 늘어나고 중복 컨텍스트가 LLM에게 전달됩니다.

### 현재 구현의 전략

`document_processor.chunk_text()`는 다음 우선순위로 동작합니다:

1. 빈 줄(`\n\n`)을 기준으로 단락 분리
2. 단락을 `chunk_size`에 맞게 탐욕적으로 합침
3. 넘치면 마지막 단락을 다음 청크의 시작으로 carry-over (overlap 역할)
4. 단락 자체가 `chunk_size`보다 길면 문자 단위로 강제 분할

### 청킹 품질 검사 체크리스트 (OCR 탭 활용)

- [ ] 가장 짧은 청크가 50자 이상인가? (너무 짧으면 정보가 없음)
- [ ] 가장 긴 청크가 chunk_size의 1.5배를 넘지 않는가?
- [ ] 청크 번호를 클릭했을 때 문장이 자연스럽게 끝나는가?
- [ ] 헤더/번호만 있는 청크가 없는가?
- [ ] 표의 내용이 연속된 하나의 청크에 담겨 있는가?

### 더 나은 청킹 전략 (향후 개선 방향)

```
현재 (고정 크기)   →   시맨틱 청킹
────────────────────────────────────
문자 수 기준으로     문단/섹션 의미 경계를
자름                 LLM으로 판단해 자름

장점: 단순, 빠름     장점: 문맥 보존
단점: 의미 경계 무시 단점: 비용, 속도
```

## 4. ③ 임베딩 모델 & 검색

### 임베딩 모델 비교

| 모델 | 차원 | 한국어 | 속도 | 권장 |
|------|------|--------|------|------|
| `bge-m3` (Ollama) | 1024 | ★★★★★ | 보통 | **1순위** — 한국어 SOTA |
| `qwen3-embedding` (Ollama) | 2048 | ★★★★ | 느림 | 고차원, 한국어 우수 |
| `nomic-embed-text` (Ollama) | 768 | ★★★ | 빠름 | 영어 중심, 비교용 |
| `all-MiniLM-L6-v2` (로컬) | 384 | ★★ | 매우 빠름 | 베이스라인 확인용 |
| `text-embedding-3-small` (OpenAI) | 1536 | ★★★★ | 빠름 | API 비용 발생 |

**한국어 CS 문서에서 기대 검색 품질**: bge-m3 > qwen3-embedding > nomic > MiniLM

### 검색 파라미터 가이드

**top_k (반환 청크 수)**

```
너무 적음 (k=1~2):  필요한 청크 누락 가능 → LLM이 "정보 없음" 응답
적정 (k=3~5):       관련 청크를 충분히 포함하면서 노이즈 최소화
너무 많음 (k=8+):   무관한 청크가 컨텍스트를 오염 → LLM 혼란, 토큰 낭비
```

현재 기본값 3 → **5로 상향 권장** (CS 질문은 여러 정책 문서에 분산된 경우 多)

**similarity_threshold (유사도 임계값)**

```
0.0 (없음):    유사도가 낮아도 무조건 top-k 반환 → 관련 없는 청크 포함 가능
0.3~0.4:       저품질 결과 필터링, 권장 시작점
0.5~0.6:       엄격한 필터 → 적절한 청크도 제외될 수 있음, threshold 이상일 때만 사용
0.7+:          매우 엄격 → 대부분 질문에서 결과 없음
```

현재 기본값 0.0 → **0.3으로 상향 권장**

### 임베딩 탭으로 검색 품질 직접 진단

1. 실행 탭 > **② 임베딩 검색** 선택
2. 실제 CS 질문 입력: `"환불 신청 방법이 어떻게 되나요?"`
3. 비교할 임베딩 모델 선택 (예: bge-m3, nomic)
4. 검색 실행 후 확인할 것:

| 체크 항목 | 좋은 신호 | 나쁜 신호 |
|-----------|-----------|-----------|
| 유사도 점수 | 상위 청크 0.6+ | 모든 청크 0.4 미만 |
| 청크 내용 | 질문에 대한 답이 포함됨 | 전혀 관련 없는 내용 |
| 모델 간 순위 | 상위 청크 순서가 유사 | 모델마다 완전히 다른 결과 |
| 반환 개수 | top-k 전부 반환 | threshold로 대부분 필터됨 |

### 하이브리드 검색 (향후 개선 방향)

현재 시스템은 **Dense(벡터) 검색만** 사용합니다.
한국어 CS에서 다음 케이스는 키워드 검색이 더 강합니다:

```
Dense가 약한 케이스:       Dense가 강한 케이스:
─────────────────────      ─────────────────────
주문번호 12345 조회        "배송이 왜 이리 느려요?"
홍길동 고객 정보            (의미 기반 매칭 필요)
"이 상품 재입고 언제?"
(특정 단어 정확 매칭)
```

하이브리드 검색 공식:
```
최종 점수 = α × dense_score + (1-α) × bm25_score
α = 0.7 (Dense 가중치) 권장 시작점
```

ChromaDB는 자체적으로 BM25를 지원하지 않으므로, 별도 BM25 라이브러리(`rank_bm25`)를 병렬로 운영해
두 점수를 합산하는 방식으로 구현할 수 있습니다.

## 5. ④ 컨텍스트 조립

### 현재 구현

`test_runner.py`:

```python
context_text = "\n---\n".join(context) if context else "참고 자료 없음"
system_prompt = _CS_SYSTEM.format(context=context_text)
```

유사도 높은 순서대로 청크를 단순 나열합니다.

### 개선 가능한 영역

**청크 출처 명시**

LLM이 어느 문서의 내용을 참고했는지 알 수 있어 답변 신뢰도와 추적성이 높아집니다.

```python
# 현재
"\n---\n".join(context)

# 개선
lines = []
for chunk, meta, score in zip(context, metadata, scores):
    lines.append(
        f"[출처: {meta['filename']}, 청크 #{meta['chunk_index']+1}, 유사도: {score:.2f}]\n{chunk}"
    )
context_text = "\n---\n".join(lines)
```

**Lost-in-the-Middle 문제**

연구 결과, LLM은 긴 컨텍스트에서 **앞부분과 뒷부분**을 더 잘 기억하고 **중간을 놓치는** 경향이 있습니다.
가장 관련도 높은 청크를 컨텍스트의 맨 앞(또는 맨 뒤)에 배치하는 것이 유리합니다.

```
현재 순서:  [1위 청크] [2위 청크] [3위 청크]
개선 순서:  [1위 청크] [3위 청크] [2위 청크]
           ↑ 앞에 배치  ↑ 뒤에 배치  ↑ 중간 (불리)
```

**중복 청크 제거**

오버랩으로 인해 연속된 두 청크가 거의 같은 내용을 포함할 수 있습니다.
코사인 유사도가 0.95 이상인 청크는 중복으로 보고 하나만 포함하는 것이 좋습니다.

## 6. ⑤ LLM 생성 & 프롬프트

### 현재 시스템 프롬프트

`test_runner.py`:

```python
_CS_SYSTEM = """당신은 농산물 직판매 쇼핑몰의 친절하고 전문적인 고객 서비스 상담원입니다.
고객의 질문에 아래 참고 자료를 활용하여 명확하고 도움이 되는 답변을 제공하세요.

[참고 자료]
{context}

참고 자료에 없는 내용에 대해서는 "정확한 안내를 위해 고객 센터(☎ 1588-0000)로 문의해 주시면 친절히 안내해 드리겠습니다." 라고 안내하세요.
반드시 한국어로 답변하세요."""
```

### 프롬프트 품질의 영향

검색이 완벽해도 프롬프트가 LLM에게 모호한 지시를 주면:
- 너무 길거나 짧은 답변
- 참고 자료를 무시하고 학습 데이터에서 답변 생성 (hallucination)
- 일관되지 않은 말투

### 개선된 프롬프트 구조

효과적인 CS 챗봇 프롬프트는 **역할 → 제약 → 형식 → 데이터** 순서로 구성합니다.

```python
_CS_SYSTEM = """역할: FarmOS 농산물 직판매 쇼핑몰 고객 서비스 AI 상담원

행동 규칙:
1. 반드시 아래 [참고 자료]에 근거해서만 답변합니다
2. 참고 자료에 없는 내용은 추측하지 말고 고객센터(☎ 1588-0000)로 안내합니다
3. 불필요한 말 없이 3~5문장 이내로 간결하게 답변합니다
4. 정중하고 친근한 말투를 사용합니다 (~습니다 / ~세요)
5. 구체적인 수치(날짜, 금액, 기한)가 있으면 반드시 포함합니다

[참고 자료]
{context}"""
```

### LLM 선택 기준

| 모델 | 강점 | 약점 | CS 챗봇 적합도 |
|------|------|------|---------------|
| Claude Haiku 4.5 | 빠름, 저렴 | 복잡한 추론 약함 | ★★★★ (속도 중요할 때) |
| Claude Sonnet 4.6 | 균형 잡힘 | 비용 中 | ★★★★★ (권장) |
| Gemini 2.0 Flash | 매우 빠름 | 한국어 일관성 낮을 수 있음 | ★★★ |
| Gemini 2.5 Flash | 빠름+정확 | 비용 中 | ★★★★ |
| llama3.2:3b (Ollama) | 무료 | 한국어 품질 낮음 | ★★ (베이스라인) |
| gemma3:4b (Ollama) | 무료 | 한국어 보통 | ★★★ |

## 7. ⑥ 평가 신뢰도

### 현재 평가 방식의 한계

현재 LLM-as-Judge 방식은 **참조 답변 없이** 채점합니다.
Judge 모델이 "그럴듯한 답변"에 높은 점수를 줄 수 있으나,
실제로 정확한 정보를 담았는지는 별도로 검증이 필요합니다.

### 평가 품질을 높이는 방법

**참조 답변(Reference Answer) 추가**

`data/default_questions.json`에서 `reference_answer` 필드를 채웁니다.

```json
{
  "category": "배송",
  "question": "주문 후 며칠 만에 배송이 오나요?",
  "expected_topics": ["배송 소요 기간", "택배사"],
  "reference_answer": "일반 택배는 결제 완료 후 영업일 기준 2~3일 이내 도착합니다. 도서산간 지역은 1~2일 추가 소요될 수 있습니다. 당일 배송은 오전 11시 이전 주문 건에 한해 제공됩니다."
}
```

Judge 프롬프트에 참조 답변을 추가하면 "형식적으로 그럴듯하지만 내용이 틀린" 답변을 걸러낼 수 있습니다.

**Judge 모델 선택**

- **빠른 프로토타이핑**: Claude Haiku (저렴, 빠름)
- **신뢰도 높은 평가**: Claude Sonnet (권장)
- Ollama 모델을 Judge로 사용하면 한국어 평가 일관성이 낮아집니다

**Judge Bias 주의**

LLM-as-Judge는 **자사 모델에 유리하게** 평가하는 경향이 있습니다.
Claude를 Judge로 쓸 때 Claude 모델의 점수가 과대 평가될 수 있습니다.
중요한 비교에서는 다른 Judge(Gemini)로 교차 검증을 권장합니다.

## 8. 진단 워크플로우

RAG 성능 문제를 진단할 때 **상위 단계부터 순서대로** 확인해야 합니다.
하위 단계를 최적화해도 상위 단계에 문제가 있으면 효과가 없습니다.

```
Step 1. OCR 탭 — 파싱/청킹 품질 확인
────────────────────────────────────────
문서 선택 후 원문 미리보기 확인:
  ✅ 텍스트가 자연스럽게 읽힘
  ✅ 표·목록 구조가 유지됨
  ✅ 페이지 번호, 헤더가 없음

청크 목록 확인:
  ✅ 청크마다 의미 있는 내용 포함
  ✅ 문장이 중간에 잘리지 않음
  ✅ 빈 청크 또는 50자 미만 청크가 없음

  ❌ 위 조건 미충족 → 파싱/청킹 수정 후 재벡터화

           ↓ 파싱/청킹 정상

Step 2. 임베딩 탭 — 검색 품질 확인
────────────────────────────────────────
실제 CS 질문 5~10개 입력 후 확인:
  ✅ 상위 청크 유사도 0.55 이상
  ✅ 반환된 청크가 질문의 답을 포함
  ✅ 관련 없는 청크가 섞이지 않음

  ❌ 유사도 낮음 → 임베딩 모델 변경 또는 청킹 재조정
  ❌ 관련 없는 청크 → similarity_threshold 설정 (0.3~0.4)
  ❌ 답 포함 청크가 없음 → 문서 내용 자체 확인 필요

           ↓ 검색 정상

Step 3. LLM 탭 — RAG OFF vs ON 비교
────────────────────────────────────────
같은 LLM으로 RAG 끄고/켜고 테스트:

  RAG ON 점수 > RAG OFF 점수:
    → 검색이 LLM에게 도움이 됨 ✅
    → LLM 모델 비교로 진행

  RAG ON 점수 < RAG OFF 점수:
    → 검색된 컨텍스트가 LLM을 방해함 ❌
    → 원인: 엉뚱한 청크 / 너무 많은 청크 / 프롬프트 문제

           ↓ RAG가 도움이 됨

Step 4. LLM 탭 — 모델 비교
────────────────────────────────────────
동일한 임베딩 모델 + 여러 LLM 비교:
결과 탭 > 리더보드에서 수치 확인

  정확성(accuracy) 낮음 → 프롬프트 개선
  관련성(relevance) 낮음 → 검색 개선
  한국어 낮음 → LLM 모델 변경 또는 프롬프트에 언어 지시 강화
  응답속도 중요 → Haiku, Flash 계열 고려
```

### 핵심 판단 기준

| 지표 | 기준선 | 해석 |
|------|--------|------|
| 임베딩 유사도 (상위 청크) | ≥ 0.55 | 정상 검색 |
| 임베딩 유사도 (평균) | ≥ 0.45 | 허용 범위 |
| 관련성 (relevance) | ≥ 7.0 | 검색 품질 양호 |
| 정확성 (accuracy) | ≥ 7.0 | 답변 신뢰 가능 |
| 종합 (overall) | ≥ 7.5 | 배포 고려 가능 수준 |
| RAG ON/OFF 차이 | ON이 +0.5 이상 | RAG가 실질적 기여 |

## 9. 우선순위별 개선 로드맵

### 🔴 즉시 적용 (1~2일, 효과 큼)

#### 1. 검색 파라미터 조정

`test_runner.py` 또는 실행 탭 설정:

```python
# 현재 기본값
top_k = 3
similarity_threshold = 0.0

# 권장 기본값
top_k = 5
similarity_threshold = 0.3
```

#### 2. 시스템 프롬프트 개선

`test_runner.py`의 `_CS_SYSTEM` 변수를 [6장 개선안](#개선된-프롬프트-구조)으로 교체.

#### 3. 임베딩 모델을 bge-m3로 우선 사용

한국어 문서에서 bge-m3가 가장 높은 검색 품질을 보입니다.
다른 모델과의 비교는 bge-m3를 기준점으로 삼아 진행합니다.

#### 4. 참조 답변 10개 추가

`data/default_questions.json`에서 각 카테고리별 대표 질문 1~2개에 `reference_answer` 작성.
정확성 평가 신뢰도가 크게 향상됩니다.

---

### 🟡 단기 개선 (3~7일, 효과 중)

#### 5. 헤더/푸터 후처리 추가

`document_processor.py`에 `_clean_text()` 함수 추가 후 `_extract_pdf` 반환 전 적용.

#### 6. 청크 출처 메타데이터 컨텍스트에 포함

#### 7. Judge 교차 검증

중요한 비교에서 Claude Haiku + Gemini 두 Judge로 각각 평가,
평균 점수를 최종 기준으로 사용.

---

### 🟢 중장기 고도화 (1~2주, 효과 높음)

#### 8. 하이브리드 검색 (BM25 + Dense)

```python
# rank_bm25 라이브러리 활용
from rank_bm25 import BM25Okapi

# Dense score와 BM25 score를 alpha로 혼합
alpha = 0.7
final_score = alpha * dense_score + (1 - alpha) * bm25_score
```

특히 주문번호, 상품명 등 키워드 기반 질문 처리에 효과적입니다.

#### 9. 리랭킹 (Cross-Encoder)

Bi-encoder(현재)로 빠르게 후보 20개 검색 후,
Cross-encoder로 정밀 재점수 매겨 상위 5개만 LLM에 전달합니다.

```
Bi-encoder: 질문 임베딩 ↔ 청크 임베딩 비교 (빠름, 덜 정확)
     ↓ 후보 20개
Cross-encoder: 질문 + 청크를 함께 입력해 상호작용 고려 (느림, 더 정확)
     ↓ 상위 5개
LLM
```

추천 모델: `ms-marco-MiniLM-L-6-v2` (영어), `cross-encoder/ms-marco-MiniLM-L6-v2`

#### 10. 문서 카테고리 기반 필터링

질문 카테고리(배송/반품/결제 등)를 자동 분류해
해당 카테고리 문서에서만 검색하면 노이즈가 크게 줄어듭니다.

```python
# ChromaDB where 필터 활용
results = collection.query(
    query_texts=[question],
    where={"category": detected_category},
    n_results=top_k,
)
```

## 10. 실험 설계 원칙

### 한 번에 하나만 바꾸기

```
❌ 잘못된 실험:
   청킹 크기 변경 + 임베딩 모델 변경 + 프롬프트 변경 → 동시에 테스트
   결과가 좋아져도 어떤 변화가 기여했는지 알 수 없음

✅ 올바른 실험:
   A/B 실험 1: 청크 300 vs 600 (임베딩·LLM 고정)
   A/B 실험 2: bge-m3 vs nomic (청크·LLM 고정)
   A/B 실험 3: Haiku vs Sonnet (청크·임베딩 고정)
```

### 충분한 질문 수 확보

5~10개의 질문으로 나온 점수 차이는 통계적으로 신뢰하기 어렵습니다.

| 질문 수 | 신뢰도 | 권장 용도 |
|---------|--------|-----------|
| 5~10개 | 낮음 | 빠른 방향 확인만 |
| 20~30개 | 중간 | 초기 모델 선정 |
| 50개+ | 높음 | 최종 의사결정 |

### 카테고리 균형

배송 질문 10개 + 결제 질문 1개로 테스트하면 배송 성능에 편향됩니다.
각 카테고리에서 균등하게 질문을 선택하거나,
**카테고리별 점수**를 별도로 확인하세요 (결과 탭 > 카테고리별 종합 점수).

### 실험 기록

각 실행에 의미 있는 이름을 붙이세요:

```
좋은 실행 이름 예시:
  "bge-m3 / chunk600 / topk5 / threshold0.3"
  "nomic / chunk300 / topk3 / RAG-OFF"
  "프롬프트v2 / bge-m3 / Sonnet"

나쁜 실행 이름 예시:
  "테스트1"
  "비교실험"
```

결과 탭의 리더보드에서 여러 실행의 점수를 나란히 비교할 때
설정이 이름에 명확히 드러나야 합니다.
