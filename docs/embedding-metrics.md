## 1. 지표 계산 방식

임베딩 탭에서 "검색 테스트 실행"을 누르면 `POST /api/debug/batch-retrieve`가 호출됩니다.  
각 임베딩 모델에 대해 20개 질문을 일괄 임베딩하고, ChromaDB에서 top-k 청크를 검색한 뒤 아래 지표를 계산합니다.

### 관련성 임계값 (Relevance Threshold)

**현재 값: 0.5** — `backend/app/routers/debug.py`의 `_RELEVANCE_THRESHOLD`에 하드코딩됩니다.

Hit@1, Hit@3, MRR 계산에서 **"관련 있음"의 기준선**으로 사용됩니다.  
"코사인 유사도 ≥ 0.5 = 이 청크는 질문과 관련 있다"고 가정합니다.

> **주의**: 이 임계값은 데이터에서 도출된 값이 아닙니다. 어떤 실험적 근거 없이 설정한 값입니다.

---

### Top-1 유사도 (avg_top1_score)

```
각 질문마다 1위로 반환된 청크의 코사인 유사도를 평균
avg_top1_score = mean([질문1의 1위 청크 점수, 질문2의 1위 청크 점수, ...])
```

"이 모델이 질문당 가장 비슷하다고 판단한 청크가 평균적으로 얼마나 가까운가"를 나타냅니다.  
높을수록 좋아 보이지만, **높은 점수 = 관련 내용** 이 아닐 수 있습니다. ([3장 참고](#3-구조적-한계))

---

### MRR (Mean Reciprocal Rank)

```
각 질문에서 "임계값(0.5) 이상인 첫 번째 청크"의 순위 역수를 평균

질문 A: 1위 청크가 0.52 → 첫 hit = rank 1 → 1/1 = 1.0
질문 B: 1위는 0.43, 2위가 0.61 → 첫 hit = rank 2 → 1/2 = 0.5
질문 C: 임계값 이상 없음 → 0
MRR = (1.0 + 0.5 + 0) / 3 = 0.5
```

1에 가까울수록 관련 청크가 항상 1위에 위치하는 모델입니다.  
0에 가까울수록 관련 청크를 하위 순위에 두거나 아예 반환하지 못하는 모델입니다.

---

### Hit@1 / Hit@3

```
Hit@1 = 1위 청크 점수가 임계값(0.5) 이상인 질문의 비율
Hit@3 = 상위 3개 청크 중 하나라도 임계값 이상인 질문의 비율
```

Hit@3 − Hit@1 차이가 크면: 관련 청크가 DB에 있지만 1위로 올라오지 못하는 경우가 많음.  
두 값 모두 작으면: 관련 청크 자체가 없거나(문서 미포함) 임베딩 품질 문제.

---

### 일관성 (score_std)

```
각 질문의 Top-1 점수들의 표준편차
낮을수록 → 질문 유형에 관계없이 비슷한 점수로 검색
높을수록 → 어떤 질문은 잘 찾고, 어떤 질문은 전혀 못 찾음
```

일관성이 낮아도 성능이 나쁜 건 아닙니다. 문서에 없는 주제를 낮은 점수로, 있는 주제를 높은 점수로 반환하는 **변별력 있는 모델**은 std가 클 수 있습니다.

---

### 순위별 하락 (rank_scores)

```
rank_scores = [1위 평균 유사도, 2위 평균 유사도, ..., K위 평균 유사도]
```

1위 → K위로 갈수록 점수가 얼마나 떨어지는지 보여줍니다.  
점수가 완만하게 떨어지면 top-K 청크 전체가 고르게 유용합니다.  
급격히 떨어지면 1위만 의미 있고 나머지는 노이즈입니다.

---

### 분포 (score_distribution)

```
Top-1 점수 기준:
  high: ≥ 0.70 (높음)
  mid:  0.50 ~ 0.70 (보통)
  low:  < 0.50 (낮음)
```

각 버킷에 몇 개의 질문이 해당하는지 보여줍니다.

## 2. 각 지표의 해석 기준

| 지표 | 좋음 | 보통 | 나쁨 |
|------|------|------|------|
| avg_top1_score | ≥ 0.65 | 0.50 ~ 0.65 | < 0.50 |
| MRR | ≥ 0.75 | 0.50 ~ 0.75 | < 0.50 |
| Hit@1 | ≥ 70% | 40% ~ 70% | < 40% |
| Hit@3 | ≥ 80% | 50% ~ 80% | < 50% |
| Hit@3 − Hit@1 | < 10%p | 10% ~ 25%p | > 25%p |
| score_std | — | 0.08 ~ 0.15 | > 0.18 |

> score_std의 "좋음" 열이 비어 있는 이유: std가 낮다고 반드시 좋은 모델이 아닙니다. 문서에 없는 주제는 낮게, 있는 주제는 높게 점수를 주는 변별력 있는 모델은 std가 클 수 있습니다. 반면 모든 쿼리에 비슷한 점수를 주는 반복 오검색 모델도 std가 낮습니다. std 단독으로는 좋고 나쁨을 판단할 수 없어 "좋음" 기준을 정의하지 않았습니다.

> 그 외 기준선들은 경험적 참고치입니다. 절대적인 기준이 아닙니다.

## 3. 구조적 한계

이 절은 지표를 **과신하기 전에 반드시 읽어야 합니다.**

### 한계 1: 정답 레이블이 없다 (가장 근본적인 문제)

현재 모든 지표는 **코사인 유사도 ≥ 0.5 = "관련 있음"** 이라는 가정에 의존합니다.

```
"배달받은 채소들을 얼마나 보관할 수 있나요?"
  → 결제/취소 청크 유사도 0.790 → Hit@1 = 1 (성공으로 카운트)
```

시스템은 반환된 청크가 질문의 답을 실제로 포함하는지 알 방법이 없습니다.  
**"유사도가 높다"는 것은 "답이 있다"와 다릅니다.**

진짜 정확도를 측정하려면 각 질문에 대해 "이 청크가 정답"이라고 레이블된 데이터가 필요합니다.  
현재 시스템에는 그런 데이터가 없습니다.

---

### 한계 2: 임계값 0.5는 근거 없는 숫자다

`_RELEVANCE_THRESHOLD = 0.5`는 코드에 하드코딩된 값입니다.  
이 숫자가 의미하려면 "유사도 0.5 이상인 청크는 실제로 관련이 있다"는 검증이 있어야 합니다.  
그런 검증은 없습니다.

더 큰 문제는 **임베딩 모델마다 점수 범위(score distribution)가 다르다**는 것입니다.

```
nomic-embed-text:    한국어 질문 → 대부분 0.65 ~ 0.82 (압축된 범위)
bge-m3:              한국어 질문 → 관련 청크 0.75+, 무관 청크 0.25~0.40 (넓은 범위)
```

같은 임계값 0.5를 적용하면:
- nomic은 거의 모든 쿼리가 "hit" → Hit@1 높게 나옴
- bge-m3는 무관 청크를 낮게 평가 → Hit@1 낮게 나올 수 있음

**변별력이 좋은 모델이 변별력이 나쁜 모델보다 지표가 낮게 나오는 역전 현상이 발생합니다.**

---

### 한계 3: 모델 간 점수 직접 비교는 불가능하다

```
bge-m3 Top-1 유사도 0.68 ≠ nomic Top-1 유사도 0.68
```

두 숫자가 같더라도 서로 다른 고차원 공간에서 계산된 값이라 직접 비교할 수 없습니다.  
모델 A의 0.7과 모델 B의 0.7이 같은 "좋음"을 뜻하지 않습니다.

---

### 한계 4: 한국어 특성이 전혀 반영되지 않는다

현재 테스트 질문은 전부 문법적으로 올바른 형태로만 들어갑니다.  
실제 CS 챗봇 사용자는 이렇게 입력합니다:

| 실제 입력 패턴 | 현재 테스트 커버 여부 |
|---------------|----------------------|
| 조사 변형: "배달" / "배달이" / "배달을" | ❌ |
| 띄어쓰기 오류: "고객서비스" vs "고객 서비스" | ❌ |
| 유의어: "교환" / "반품" / "환불" | ❌ |
| 구어체: "이거 어떻게 해요?" | ❌ |
| 오탈자: "배솔", "빼달" | ❌ |

이런 변형에 강건한 모델이 실제 서비스에서 더 잘 작동하지만, 현재 지표는 이를 측정하지 않습니다.

---

### 한계 5: 청킹 방식이 한국어 형태소를 고려하지 않는다

현재 청킹은 문자 수(char count) 기준으로 단락을 분리합니다.  
한국어는 **교착어**로, 하나의 어근에 조사·어미가 붙어 단어 형태가 달라집니다.

```
"환불" / "환불이" / "환불을" / "환불하다" / "환불되다"
→ 문자 수 기준 청킹: 모두 다른 문자열로 처리
→ 형태소 분석: 모두 같은 어근 "환불"로 처리 가능
```

임베딩 모델 자체가 이 변형을 흡수할 수 있다면 문제가 없습니다.  
그러나 영어 중심 모델이나 학습 데이터가 충분하지 않은 모델에서는,  
청킹 경계가 형태소 경계를 무시하고 단어를 중간에서 잘라  
임베딩이 의미 없는 부분 문자열을 처리하게 될 수 있습니다.

> **이 문서의 범위**: 현재 지표는 이 문제를 감지하거나 측정하지 않습니다.  
> 임베딩 모델을 교체해도 청킹이 한국어 형태소를 고려하지 않으면 검색 품질의 상한이 낮아질 수 있습니다.  
> 해결 방향(형태소 분석기 기반 청킹 등)은 [RAG 성능 개선 가이드](./rag-performance-guide.md)를 참고하세요.

---

### 한계 6: 문서에 없는 내용을 물어보는 테스트가 없다

좋은 임베딩 모델은 문서에 없는 질문을 받았을 때 낮은 유사도 점수를 반환해야 합니다.  
현재 테스트 질문 20개는 전부 FarmOS CS 관련 질문이라,  
"문서 범위 밖 질문에 낮은 점수를 주는 능력"은 전혀 측정하지 않습니다.

## 4. 반복 오검색 감지

시스템에서 자동으로 감지하는 패턴입니다.  
학술적으로는 "Hubness Problem" (Radovanović et al., JMLR 2010)으로 알려진 현상입니다.

### 무엇인가

특정 청크가 서로 무관한 여러 쿼리의 Top-1으로 반복 반환되는 현상입니다.

```
"배달받은 채소 보관 기간?" → 결제/취소 청크 0.790
"배송 추적은 어떻게 해요?" → 결제/취소 청크 0.762
"교환 신청 방법이 뭐예요?" → 결제/취소 청크 0.741
"포인트 사용 조건이 있나요?" → 결제/취소 청크 0.755
```

### 왜 발생하는가

한국어 중심으로 훈련되지 않은 모델(nomic-embed-text 등)은  
한국어 텍스트를 임베딩 공간의 좁은 영역으로 압축합니다.  
그 공간 안에서 "자주 나오는 단어를 많이 포함한 청크"가 중심에 위치해  
어떤 쿼리와 비교해도 비슷한 거리(높은 유사도)를 갖게 됩니다.

### 지표 왜곡 효과

| | 반복 오검색 없는 모델 | 반복 오검색 있는 모델 |
|---|---|---|
| 실제 검색 품질 | 좋음 | 나쁨 |
| Hit@1 지표 | 낮게 나올 수 있음 | 높게 나옴 (부풀려짐) |
| MRR 지표 | 낮게 나올 수 있음 | 높게 나옴 (부풀려짐) |
| avg_top1_score | 낮게 나올 수 있음 | 높게 나옴 (부풀려짐) |

즉, **반복 오검색이 있는 모델이 없는 모델보다 지표가 더 좋게 보이는 역전**이 발생합니다.

### 감지 기준

전체 질문의 20% 이상에서 동일한 청크가 Top-1으로 반환될 경우 경고합니다.  
감지 시 UI에 "반복오검색 N%" 뱃지가 표시되고, Hit@1(조정) 값이 함께 표시됩니다.

**동일 청크 판별 방식**: 청크 텍스트의 앞 120자를 키로 사용합니다.

- **120자로 고정한 이유**: 청크 크기 설정(기본 600자)과 무관하게 안정적으로 작동하도록 고정값을 사용합니다. 청크 시작 부분은 내용을 가장 잘 대표하며, 120자면 대부분의 한국어 청크에서 고유한 식별자 역할을 합니다.
- **한계**: 청크 크기를 300자 이하로 줄이면 두 청크가 실제로는 다른 내용임에도 앞 120자가 같을 확률이 높아집니다. 반대로 청크가 매우 길면(1000자+) 앞 120자만으로 충분히 구분됩니다. 현재 기본 청크 크기(600자) 기준으로는 적절하게 동작합니다.
- **임계 비율 20%**: 전체 질문 수의 20% 이상이 같은 청크를 Top-1으로 받는다면 우연으로 보기 어렵다는 경험적 기준입니다. 테스트 질문 20개 기준으로는 4개 이상이 같은 청크를 반환할 때 경고합니다.

**Hit@1(조정)**: 반복 청크가 Top-1인 쿼리를 miss로 처리하고 재계산한 값.  
이 값이 원본 Hit@1보다 훨씬 낮다면 해당 모델의 원본 지표는 신뢰하기 어렵습니다.

## 5. 지표를 어떻게 읽어야 하는가

### 절대값보다 상대 비교

MRR 0.73이 "좋은가 나쁜가"는 단독으로 판단할 수 없습니다.  
같은 질문 셋, 같은 문서에서 다른 모델과 비교했을 때의 **순서**가 의미 있습니다.

### 반복 오검색 여부를 먼저 확인

"반복오검색" 뱃지가 표시된 모델은 지표 전체를 신뢰하기 어렵습니다.  
Hit@1(조정) 값을 참고하고, 해당 모델의 고유사도 케이스 목록을 직접 열어 내용을 확인하세요.

### 점수가 아니라 청크 내용을 직접 확인

분석 탭의 "고유사도 케이스" 섹션에서 실제 반환된 청크 내용을 직접 읽으세요.  
숫자가 좋아도 내용이 엉뚱하면 그 모델은 이 문서에 적합하지 않은 겁니다.

### Hit@3 − Hit@1 차이에 주목

차이가 크다면 관련 청크가 1위에 올라오지 못하고 2~3위에 있는 경우입니다.  
검색 자체는 가능하지만 순위 품질에 문제가 있다는 신호입니다.  
top-k를 늘리거나 청크 크기를 줄이면 개선될 수 있습니다.

---

## 6. 실제로 신뢰할 수 있는 평가 방법

현재 임베딩 지표의 한계를 보완하는 방법들입니다.

### 방법 1: LLM 답변 품질을 임베딩 평가 대용으로 활용 (현재 시스템에서 가능)

**원리**: 검색 품질이 좋으면 → LLM이 좋은 컨텍스트를 받음 → 답변 품질 점수가 높아짐

이미 실행(runs)에서 LLM 답변에 대한 평가 점수(relevance, accuracy, helpfulness 등)가 생성됩니다.  
임베딩 모델별로 이 점수들을 묶어 평균내면 됩니다.

```
임베딩 모델 A + LLM X → accuracy 평균 6.2
임베딩 모델 B + LLM X → accuracy 평균 7.8
                       ↑ 이 차이가 임베딩 품질 차이
```

동일한 LLM을 고정하고 임베딩 모델만 바꿔서 실행했을 때 답변 점수가 더 높은 임베딩이 더 나은 임베딩입니다.  
결과 탭 > "임베딩별 집계" 뷰에서 이 비교가 가능합니다.

---

### 방법 2: 정답 청크 레이블 추가

`backend/data/default_questions.json`에 `reference_chunk_keywords` 필드를 추가합니다.

```json
{
  "category": "환불",
  "question": "환불 신청 기한이 어떻게 되나요?",
  "reference_chunk_keywords": ["환불", "7일", "수령 후"]
}
```

청크를 반환받았을 때 해당 키워드가 포함된 청크를 "정답 청크"로 간주하면  
임계값 없이 실제 Precision@K를 계산할 수 있습니다.  
이 방식은 5~10개 질문에만 적용해도 충분히 유용한 신호가 됩니다.

---

### 방법 3: 조사 변형 질문으로 한국어 강건성 테스트

같은 질문을 여러 형태로 입력해 일관된 청크를 반환하는지 확인합니다.

```
기본형:    "배송 기간이 얼마나 되나요?"
조사 변형: "배송 기간은 얼마나 됩니까?"
구어체:    "배송 얼마나 걸려요?"
띄어쓰기: "배송기간 얼마나되나요?"
```

네 질문 모두 동일한 청크를 1위로 반환하면 강건성이 높은 모델입니다.  
결과가 다르면 조사/띄어쓰기 변형에 취약한 모델입니다.

---

### 방법 4: 무관한 질문 포함 (Out-of-Domain 테스트)

문서 주제와 전혀 무관한 질문을 추가해 낮은 점수를 주는지 확인합니다.

```json
{
  "category": "OOD테스트",
  "question": "비트코인 현재 가격이 얼마예요?",
  "note": "이 질문에 대해 모든 청크가 낮은 유사도를 반환해야 함"
}
```

좋은 임베딩 모델은 무관한 질문에 0.2~0.3대의 낮은 점수를 반환합니다.  
반복 오검색이 있는 모델은 무관한 질문에도 0.7+ 점수를 반환합니다.

---

### 방법 5: 백분위 기반 동적 임계값으로 모델 간 공정 비교

[3장 한계 2](#한계-2-임계값-05는-근거-없는-숫자다)에서 설명했듯, 고정 임계값 0.5는 점수 분포가 압축된 모델에 유리하고 변별력 있는 모델에 불리합니다. 이를 보완하려면 **각 모델의 점수 분포를 먼저 분석해 모델별로 임계값을 조정**해야 합니다.

**원리: 고정 임계값 대신 백분위 기반 임계값**

```
1. 각 모델의 Top-1 점수 분포를 수집
   nomic: [0.71, 0.74, 0.79, 0.75, 0.68, ...]
   bge-m3: [0.82, 0.31, 0.77, 0.28, 0.85, ...]

2. 각 모델에서 상위 N%에 해당하는 점수를 임계값으로 사용
   예: 상위 50%(중앙값)를 임계값으로 설정
   nomic 동적 임계값 ≈ 0.73
   bge-m3 동적 임계값 ≈ 0.54

3. Hit@1 재계산
   → 각 모델의 임계값은 다르지만 "이 모델에서 상위 절반인가"라는
     공통된 기준으로 비교 가능
```

이 방식을 쓰면 "모든 쿼리를 0.7 이상으로 반환하는 모델"과 "관련 쿼리만 0.8, 나머지는 0.2로 반환하는 모델"을 같은 척도로 평가할 수 있습니다.

**현재 시스템 적용 방법**

`backend/app/routers/debug.py`의 `_compute_aggregates()` 함수에서 고정 임계값을 동적 임계값으로 교체하면 됩니다.

```python
# 현재: 고정 임계값
_RELEVANCE_THRESHOLD = 0.5

# 개선: 중앙값 기반 동적 임계값
import statistics
dynamic_threshold = statistics.median(top1_scores) if top1_scores else 0.5
```

단, 동적 임계값을 사용하면 절대적인 "이 모델의 Hit@1이 N%다"라는 의미가 바뀝니다.  
모델 내 상대 순위를 기준으로 삼는 것이므로, 결과 해석 시 "동적 임계값 사용"임을 명시해야 합니다.

**주의**: 동적 임계값은 비교 공정성을 높이지만, "이 모델이 실제로 좋은가"를 말해주지는 않습니다. 정답 레이블이 없는 한계(한계 1)는 여전히 남습니다.

## 요약: 시스템 현재 구현 상태

아래 표는 이 문서에서 다룬 평가 기능들이 **현재 시스템에 어떻게 구현되어 있는지**를 정리한 것입니다.  
문서의 한계나 이상적인 목표가 아니라, 지금 코드베이스의 실제 상태입니다.

| 평가 기능 | 구현 상태 | 관련 챕터 |
|-----------|-----------|-----------|
| 정답 레이블 기반 Precision@K | ❌ 미구현 — 유사도 임계값으로 대체 중 | 3장 한계 1 |
| 임계값 근거 | ❌ 없음 — 0.5는 임의 설정값 | 3장 한계 2 |
| 백분위 기반 동적 임계값 | ❌ 미구현 — 고정값 사용 중 | 6장 방법 5 |
| 모델 간 점수 정규화 | ❌ 미구현 — 원점수 직접 비교 | 3장 한계 3 |
| 반복 오검색 자동 감지 | ✅ 구현됨 — 지표 왜곡 자동 경고 + 조정 Hit@1 | 4장 |
| 한국어 변형 강건성 테스트 | ❌ 미구현 — 정형 질문만 사용 | 3장 한계 4·5 |
| OOD(범위 밖) 질문 테스트 | ❌ 미구현 | 3장 한계 6 |
| LLM 답변 품질과의 연계 분석 | ⚠️ 데이터는 생성됨 — 임베딩별 집계 뷰 미구현 | 6장 방법 1 |

**결론**: 현재 임베딩 지표는 모델 간 **경향성 비교**에는 유용하지만, 수치 자체를 "성능의 절대값"으로 해석하면 안 됩니다. 가장 신뢰도 높은 평가는 동일 LLM + 임베딩 모델만 바꿔서 실행한 뒤 LLM 답변 품질 점수를 비교하는 것입니다.
